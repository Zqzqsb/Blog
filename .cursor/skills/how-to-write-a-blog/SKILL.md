# 技术博客写作

## 何时遵循

在生成或改写**技术向长文**（架构解析、机制说明、对比选型）时应用下列约束；与主题无关的短文不必强行套用。

## 写作目标

- **搭建心智模型**：默认读者已经在用某项技术，文章重点是帮他理清结构和关键路径，而不是从零教学。
- **解释设计与取舍**：不仅写「是什么 / 怎么做」，还要点出「为什么这么设计、在解决什么问题、放弃了什么」。

## 语气与风格

- **中性、事实导向**：
  - 用「X 做…，原因是…」、「在实践中通常…」这类客观表述。
  - 避免第二人称教学口吻：少用「你应该…、我们来看看…、手把手教你…」。
- **克制的评论**：
  - 可以简短评价（如「这个方案代价是…」），但不搞情绪化渲染或段子。
- **精简但不省略关键链路**：
  - 宁可删示例或冗长代码，也不要省略关键的调用关系 / 数据流。

## 结构与分段

- 开头一小段：**一句话定义 + 为什么值得关心**。
- 正文按**层次 / 阶段 / 组件**拆分，而不是按「安装 / 配置 / 教程步骤」拆。
- 每个主要章节开头可以用一行引用 `>` 概括该节核心观点（1–2 句话）。
- **章节粒度**：
  - 一级标题：整体架构或大模块。
  - 二级标题：子模块或关键路径（协议、线程模型、缓存、写路径等）。
  - 三级标题只在必要时用来区分并列形态（例如：三种客户端、几种实现方式）。

## 内容取舍优先级

**优先解释：**

- **边界 / 接口**：谁调用谁、依赖谁（如 CLI → Driver → 协议 → Server → Engine）。
- **数据 / 控制流**：一条请求在各层之间如何流动。
- **问题 → 设计 → 机制**：
  - 先说遇到的问题 / 瓶颈；
  - 再说设计思路；
  - 最后解释具体实现机制。

**可以弱化或删掉：**

- 过多配置项表格（只保留真正影响行为 / 性能的）。
- 冗长的完整代码示例（只保留能看出调用链的最小片段）。
- 与主题关系弱的「顺手技巧」。

## 图、表、代码

- **图优先于长段说明**：
  - 用 Mermaid / 简图表达：架构分层、协议时序、数据流、演进路线等。
  - 图下用 1–3 条 bullet 解释「这图想表达什么」。
- **表格用于并列比较**：
  - 典型场景：多种模式（异步 / 半同步 / 强一致）、多种方案（Proxy vs Agent）、多种实现方式等。
  - 列设计：模式 / 机制 / 一致性或优缺点 / 代价。
- **代码片段的职责**：
  - 展示**调用链和关键点**，不是完整项目样例。
  - 例子中要清楚体现「谁在上层、谁在包装谁」（如 ORM → 连接池 → Driver）。

## 解释「谁实现了什么」

- 对所有成组出现的名词，明确：
  - 谁是**官方组件**（如官方 binary / driver / server）。
  - 谁是**第三方库 / ORM / 中间件**。
  - 它们之间是**静态链接 / 动态链接 / 协议对接**的关系。
- 用一行调用链或简单表格说明：`A (应用/ORM) → B (连接池) → C (Driver) → 协议 → Server`。

## 设计动机与代价

对每个关键机制（连接池、线程池、WAL、复制模式、Proxy / Agent、缓存、多副本一致性等）都遵循：

- **因为**：原始问题 / 痛点 / 瓶颈是什么；
- **所以**：设计思路（抽象层级、解耦、事件驱动、共识协议…）；
- **具体怎么做**：简述实现要点；
- **代价和适用场景**：什么时候值得用，什么时候成本过高。

## 不需要的东西

- 不写冗长的「安装步骤 / Hello World 教程」，除非与架构或设计点强相关。
- 不写营销/标题党式文案和鸡汤段落。
- 不写没有前提条件的「万能最佳实践清单」，所有建议都要给出**前提和适用范围**。
