> C++11 引入的 `std::atomic` 是并发编程的基础设施，但 `load`、`fetch_add`、`fetch_sub` 只是 API 层面，真正的陷阱在 `memory_order`。本文用"围栏"直觉解释 acquire/release/acq_rel/relaxed 四种内存序的区别，以及为什么正确的原子操作需要正确的顺序约束。

---

## 1. 原子操作三件套

### 1.1 `load()` — 原子读

```cpp
int32_t val = ref_count_.load(memory_order_acquire);
```

从内存中原子地读取当前值。"原子"意味着不会读到写了一半的中间态（比如 32 位值的高 16 位是新的、低 16 位是旧的）。

等价于普通读 `int32_t val = ref_count_`，但：
- 普通读在编译器/CPU 眼里可以被任意重排
- `.load(acquire)` 带一个获取屏障，阻止后续读写被提前到此读之前

### 1.2 `fetch_add(1)` — 原子读-改-写（加）

```cpp
ref_count_.fetch_add(1, memory_order_acq_rel);
```

一条指令完成三步：读旧值 → +1 → 写回新值，中间不会被其他线程插入。

如果两个线程同时 `fetch_add(1)`，结果一定是 +2，不会丢失。这就是"原子"的意义——没有 race。

返回值是**修改前的旧值**（虽然示例中没用到）。

### 1.3 `fetch_sub(1)` — 原子读-改-写（减）

```cpp
ref_count_.fetch_sub(1, memory_order_acq_rel);
```

和 `fetch_add` 完全对称，只是做减法。同样返回修改前的旧值。

## 2. 为什么不用普通 `++` / `--`？

```cpp
// 危险：非原子操作
ref_count_++;  // 实际是 tmp = ref_count_; tmp = tmp + 1; ref_count_ = tmp;
```

这三步之间如果另一个线程也在 `++`：

```
线程A: tmp_a = 5
线程B: tmp_b = 5          ← 读到同一个旧值
线程A: ref_count_ = 6
线程B: ref_count_ = 6     ← 覆盖了 A 的结果，丢了一次加
```

`fetch_add` 是 CPU 级别的单条指令（x86 上是 `lock xadd`），硬件保证不会出现这个问题。

## 3. `memory_order` 的直觉：围栏

原子操作本身是正确的，围栏控制的是它旁边的普通读写能不能被 CPU/编译器搬到围栏的另一边：

### acquire（获取围栏）

```
                    acquire
                    ─────────────────
普通读写不能越过     ↑ 这条线往上跑
                    ↓ 可以往下跑
```

`.load(memory_order_acquire)` 保证：**后续的读写不会被重排到这次读之前**。

### release（释放围栏）

```
                    release
                    ─────────────────
普通读写不能越过     ↓ 这条线往下跑
                    ↑ 可以往上跑
```

`.store(x, memory_order_release)` 或带 release 语义的写操作保证：**之前的读写不会被重排到这次写之后**。

### acq_rel = 两道围栏叠加

```
                    acq_rel
                    ─────────────────
                    ↑ 上面的不能下来
                    ↓ 下面的不能上去
```

读-改-写操作（如 `fetch_add`）同时带 acquire 和 release 语义：
- 作为"读"阻止后续操作提前
- 作为"写"阻止前面操作延后

### relaxed = 没有围栏

CPU 随便重排。只保证原子性，不保证顺序。

## 4. acquire/release 配对：建立 happens-before

```cpp
// 线程 A: 写完数据后 inc
node->vec_ = new_vec;              // ①
node->neighbors_ = new_neighbors;  // ②
ref_count_.fetch_add(1, memory_order_acq_rel);  // ③
                                   // ↑ release 围栏：①② 不能跑到 ③ 后面

// 线程 B: 看到 ref_count > 0 后读数据
int rc = ref_count_.load(memory_order_acquire); // ④
                                   // ↑ acquire 围栏：⑤⑥ 不能跑到 ④ 前面
if (rc > 0) {
    use(node->vec_);               // ⑤ ← 保证看到 ① 的结果
    use(node->neighbors_);         // ⑥ ← 保证看到 ② 的结果
}
```

③ 的 release 和 ④ 的 acquire 配对，形成一个跨线程的 **happens-before** 关系：

```
① ② → ③ → ④ → ⑤ ⑥
```

如果 ③ 用 `relaxed`，CPU 可能把 ③ 排到 ① 前面执行，线程 B 看到 `ref_count_ == 1` 时 `vec_` 可能还没写完。

## 5. 常见误读

**"`memory_order_seq_cst` 最安全，一律用它"**：seq_cst 确实最强，但开销最大（需要全局同步屏障）。大多数场景用 `acquire`/`release`/`acq_rel` 已经足够，relaxed 只在完全不需要顺序保证时使用（如纯计数器）。

**"原子变量用了默认序就万事大吉"**：C++11 原子操作的默认序是 `seq_cst`，但显式指定 `relaxed` 或错误使用 `acquire`/`release` 而不配对，仍会导致数据竞争。

**"`volatile` 可以替代 atomic"**：`volatile` 只禁止编译器优化，不保证原子性，也不提供内存序控制。多线程场景必须用 `std::atomic`。
